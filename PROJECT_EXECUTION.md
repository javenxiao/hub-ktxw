# Project Execution Guide

本文件用于指导本项目在 8 周内以 AI 驱动方式推进开发。

适用前提：

- 你是项目负责人和验收人
- 你不自己写代码
- AI 负责实现、修复、重构、测试、文档更新
- 你负责拆任务、定优先级、做设备联调、做验收决策

## 1. 使用原则

如果你希望在 1 到 2 个月内完成项目，并且自己不写代码，必须坚持下面的协作方式：

1. 每次只给 AI 一个小任务，控制在半天到一天内完成。
2. 每个任务都必须写清楚目标、允许修改的文件、不能破坏的约束、验收标准、必须执行的验证。
3. 严禁让 AI 一次处理“整个 SDK 接入”“整个页面重构”“整个项目优化”。
4. 每次任务结束后，AI 必须执行构建、运行或接口验证，并明确说明剩余风险。
5. 必须始终保住两种运行模式：
   - 模拟模式可运行
   - 真实设备模式可接入
6. 硬件联调问题和页面问题分开处理，不要混在一个任务里。
7. 你每天只做 4 件事：发任务、看结果、做设备联调、判定是否验收通过。

## 2. 8 周总目标

8 周内的交付目标应控制为：

1. Web 页面稳定运行，模拟模式始终可展示。
2. 真实图传 SDK 生命周期完整接入，包括设备枚举、打开、初始化、启动、停止、关闭。
3. 首批真实字段替换完成，包括无线基础信息、流量统计、连接状态、RSSI 曲线。
4. 出现第三方依赖或设备异常时，服务可回退并输出可排查日志。
5. 项目具备可交付文档、调试文档和验收清单。

## 3. 周目标总览

### Week 1

目标：冻结范围，补齐字段映射和 SDK 能力映射，建立统一验收口径。

### Week 2

目标：补齐设备生命周期相关绑定和 Rust 封装，让真实设备会话具备完整入口。

### Week 3

目标：建立真实数据采集层和页面专用转换层，先替换 General 区域字段。

### Week 4

目标：补齐 socket 生命周期和统计能力，替换 Traffic 区域。

### Week 5

目标：接入连接状态相关真实字段，替换 Connection 区域。

### Week 6

目标：接入 RSSI 历史采样和曲线，替换 Chart 区域。

### Week 7

目标：补齐调试接口、日志、依赖检查和诊断能力。

### Week 8

目标：完成回归、文档、部署说明和交付验收。

## Week 1: 范围冻结与基线建立

### W1-T1 基线构建与运行验证

- 建议时长：0.5 天
- 目标：确认 cargo build、cargo run、模拟模式、页面加载、基础接口均可用。
- 建议修改文件：README.md、.vscode/tasks.json、必要的少量运行脚本或日志说明文件。
- 验收标准：
  - cargo build 成功
  - cargo run 成功
  - 页面可打开
  - /api/system/info 和 /api/wireless/status 返回正确 JSON
  - SDK 不可用时能自动回退到模拟模式

### W1-T2 页面字段与接口字段映射表

- 建议时长：0.5 天
- 目标：梳理当前页面 DOM 字段、接口 JSON 字段、Rust 结构体字段之间的对应关系。
- 建议修改文件：README.md 或单独新增一份字段映射文档。
- 验收标准：
  - System、Wireless、Traffic、Connections、Chart 各区字段全部列出
  - 标注哪些字段仍是模拟值
  - 标注每个字段未来应来自哪个后端结构

### W1-T3 SDK 头文件能力映射表

- 建议时长：1 天
- 目标：从 bb_api.h 和 bb_config.h 中梳理设备生命周期、ioctl、socket、调试能力，并形成后续开发映射表。
- 建议修改文件：README.md 或单独新增一份 SDK 能力文档。
- 验收标准：
  - 列出设备枚举、设备打开关闭、init/start/stop、ioctl、socket、debug 能力
  - 标注每类能力在 Web 项目中的用途
  - 标注后续优先接入顺序

## Week 2: 设备生命周期与 Rust 封装

### W2-T1 补齐设备生命周期 FFI 绑定

- 建议时长：1 天
- 目标：在 FFI 层补齐 bb_dev_getlist、bb_dev_open、bb_dev_close、bb_start、bb_stop、bb_ioctl、bb_ioctl_ex 等必要绑定。
- 建议修改文件：src/ffi.rs。
- 验收标准：
  - 新增绑定与头文件签名一致
  - 不凭猜测命名函数或参数
  - cargo build 成功

### W2-T2 重构 BasebandManager 的会话模型

- 建议时长：1 天
- 目标：将当前仅有初始化功能的 BasebandManager 重构为带设备句柄和会话状态的管理器。
- 建议修改文件：src/bb_api.rs、src/main.rs，必要时允许新增 src 内部模块。
- 验收标准：
  - 区分 SDK 初始化状态、设备打开状态、start 状态、socket 状态
  - 不把裸句柄逻辑散落在 main.rs
  - cargo build 成功

### W2-T3 新增 SDK 健康状态接口

- 建议时长：0.5 天
- 目标：新增一个健康状态接口，向页面或调试工具暴露 SDK 是否加载成功、设备是否打开、当前是否处于模拟模式。
- 建议修改文件：src/main.rs、src/bb_api.rs。
- 验收标准：
  - 新增只读 JSON 接口
  - 返回结构清晰可读
  - 模拟模式与真实模式状态可区分

## Week 3: 真实采集层和 General 区域替换

### W3-T1 补齐 ioctl 包装层

- 建议时长：1 天
- 目标：把常用 request code 和 ioctl/ioctl_ex 调用封装成可复用的 Rust 方法。
- 建议修改文件：src/ffi.rs、src/bb_api.rs。
- 验收标准：
  - request code 不直接散落在 main.rs
  - Rust 层统一返回 Result
  - 失败时有清晰错误信息

### W3-T2 建立原始结构和页面转换结构

- 建议时长：1 天
- 目标：引入原始采集结构和页面专用结构的转换层，避免 SDK 原始值直接灌进页面 JSON。
- 建议修改文件：src/main.rs、src/bb_api.rs，必要时允许新增 src/models.rs 或 src/mapper.rs。
- 验收标准：
  - RawStatus 和页面结构分离
  - 至少有一个显式转换函数
  - 页面字段命名不直接暴露 SDK 内部命名

### W3-T3 替换 General 区域为真实值

- 建议时长：1 天
- 目标：将频率、带宽、发射功率、模式等 General 区域字段，优先替换为真实值或半真实值。
- 建议修改文件：src/main.rs、src/bb_api.rs、static/index.html。
- 验收标准：
  - 页面可看到真实字段变化
  - 真实值失败时仍能回退默认值
  - 接口 JSON 与页面显示一致

## Week 4: Socket 生命周期与 Traffic 区域替换

### W4-T1 完成 socket fd 生命周期管理

- 建议时长：1 天
- 目标：在 Rust 层明确 socket 的打开、关闭、重连和异常回收逻辑。
- 建议修改文件：src/bb_api.rs、src/ffi.rs。
- 验收标准：
  - socket fd 生命周期清晰
  - 不把 socket id 当作固定常量到处硬编码
  - 失败时有明确日志

### W4-T2 完成流量统计获取逻辑

- 建议时长：1 天
- 目标：基于 socket ioctl 或应用层累计，补齐发送字节数、接收字节数、收发统计的真实采集逻辑。
- 建议修改文件：src/bb_api.rs、src/ffi.rs、src/main.rs。
- 验收标准：
  - 统计值来源明确
  - 接口返回值非纯模拟
  - 失败时不会导致页面崩溃

### W4-T3 替换 Traffic 区域为真实值

- 建议时长：0.5 天
- 目标：把 Traffic 区域从模拟数据切换到真实采集结果。
- 建议修改文件：src/main.rs、static/index.html。
- 验收标准：
  - 页面显示真实收发统计
  - WebSocket 或接口刷新逻辑正常
  - 模拟回退逻辑仍可用

- 验收标准：
  - 明确连接状态字段来源
  - 明确 MAC、SNR、调制方式、RSSI 的获取路径

### W5-T2 实现连接状态采集逻辑

- 建议时长：1 天

- 验收标准：
  - 至少接入对端 MAC、SNR、主副 RSSI 中的一部分真实值
  - 失败时不影响接口整体可用性

## 5. 每日 AI 使用方式

建议你的日常节奏固定为：

1. 每天早上只选 1 个主任务和 1 个备选任务。
2. 把任务填进下面的 AI 工单模板，直接发给 AI。
3. AI 做完后，你只做 3 件事：
   - 看构建是否通过
   - 看页面或接口是否符合预期
   - 做真实设备联调或异常复现
4. 如果当天的主任务被硬件阻塞，立刻切到备选任务，不要卡死。
5. 每天下班前让 AI 输出“已完成、阻塞点、明天第一优先级”。

## 8. 你在整个项目中的角色

为了避免你重新陷入“自己动手写代码”，你要强制把自己的角色限制为：

1. 任务拆分者
2. AI 指挥者
3. 设备联调者
4. 页面验收者
5. 风险判断者

你不要做的事：

1. 自己手改实现代码
2. 自己补 FFI 绑定
